## Current Shape

The codebase now has a working control-plane skeleton:

- Go gateway, control worker, node worker, and local CLI.
- PostgreSQL migrations with users, projects, project templates, repositories, git remotes, operation runs, worker jobs, AI runtime/task skeletons, SSH machines, Argo connections, deployment targets, deployment records, rollback points, adapter run tables, and the first Asset/AssetRelation tables.
- React/Ant Design console for dashboard, assets, projects, git remotes, operations, workers, AI runtime/tasks, Argo, and SSH.
- Real adapters for Git sync/tag, GitHub Actions sync, Argo app sync, and SSH command execution.
- Durable approval requests for high-risk operations and operator-only SSH command output.
- Context file generation for `.assops/context`.
- Derived asset inventory APIs that expose current projects, project templates, repositories, remotes, operation runs, pipeline runs, hosts, Argo apps, deployment targets, runtimes, agent tasks, and node agents through a common asset shape.

This is stronger than a mock-only skeleton, but it is not yet the asset-centered product described in Notion.

## Readiness Gate Status

The Dashboard and `assops-tool project readiness` now use the same first-version readiness contract, and all ten demo gates require canonical asset or graph evidence instead of loose aggregate rows:

- Project asset: canonical `project` asset plus `project:*` graph node.
- Repository attachment: same repository linked from project to at least two Git remotes.
- RepoSyncAsset: repository-to-sync, sync-to-source-remote, and sync-to-target-remote graph links.
- Manual and webhook sync: Gitea webhook connection and event linked through the same RepoSyncAsset and operation run.
- GitHub Tag and Actions: project-to-repository-to-remote-to-action-run graph chain plus completed/succeeded/success tag operation linked to a project-owned remote.
- SSH audit: host asset evidence, first-class `ssh.verify` operation evidence, SSH command operation evidence, and at least two operation-to-command-run-to-machine graph audit chains plus command-run asset evidence.
- Argo deployment view: connection-to-app-to-target graph chain plus sync-operation evidence.
- Operation history: canonical operation-run asset plus operation log evidence.
- Approval policy: approval request asset, active approval-rule asset, and rule-to-approval `governs` relation.
- AI context: graph evidence plus agent-task-to-runtime and agent-task-to-`context.generate` tool-call links.

This makes the current gap measurable: a first-version demo is ready only when the live environment can satisfy those ten graph-backed gates, not merely when individual tables contain sample rows.

## Notion Target

The product direction is an engineering asset management platform, not a CI/CD-only tool. The core abstractions are:

- Unified asset ledger.
- Asset relation graph.
- Operation engine with approval and audit.
- RepoSync as a first-class asset.
- SSH worker based Gitea to GitHub sync.
- AI Agent that acts only through controlled assets and operations.

The staged plan in Notion starts with an asset ledger, then provider sync, then Repo Sync SSH, then project template instantiation, deployment, and AI Agent.

### 2. RepoSyncAsset Lifecycle Is Still Narrow

Notion treats RepoSyncAsset as the key model for Gitea -> GitHub SSH sync. The implementation now has `repo_sync_assets`, list/create/run/detail/update/archive/restore APIs, worker status tracking, a web UI for saving/running/editing/enabling/disabling/archiving/restoring source -> target sync policies, failed sync rerun from the last ref, RepoSync run history filters, per-asset run analytics (total, success/failure/running counts, success rate, recent failures, average duration, list-level risk summaries, 14-day trend, and provider/capacity signals), and a Gitea push webhook path that can enqueue enabled webhook-triggered assets. Archive currently means hidden, disabled soft-archive; restore clears the archive marker and re-enables the asset.

Impact:

- RepoSyncAsset lifecycle basics are now covered, and webhook-triggered sync policies now have replay and one-time secret rotation controls.
- Operators can inspect per-asset sync health, list-level risk summaries, recent trend, active queue pressure, provider-pair pressure, provider status, webhook failures, GitHub Actions volume/status, and the warning/danger thresholds behind each capacity signal tied to the selected repository and project without opening every individual run. Webhook connection delivery failures also flow into canonical asset health/status snapshots for Asset Center and AI context review.
- Webhook shared secrets are now encrypted at rest for new/rotated connections, with legacy plaintext fallback for existing rows.
- The UI now shows a copyable fully qualified webhook URL generated from `ASSOPS_GATEWAY_URL`.
- GitHub `workflow_run` webhooks can now update the `github_action_runs` read model for the connected GitHub remote.

Recommended next slice:

- Tune provider-pair capacity thresholds after real sync volume and provider limits are known; the current thresholds are now named in code and visible in the RepoSync detail signal table.

### 3. Webhook Intake Needs Hardening

Notion's critical flow is Gitea push webhook -> Gateway -> Worker SSH sync -> GitHub Actions. Current implementation supports project-scoped `webhook_connections`, encrypted-at-rest one-time shared secrets for new/rotated connections, HMAC-SHA256 verification, `POST /api/webhooks/gitea/{connection_id}`, `POST /api/webhooks/github/{connection_id}`, Gitea push ref parsing, matching enabled RepoSyncAssets, enqueueing `repo.sync`, GitHub `workflow_run` read-model updates, webhook event audit rows, webhook event canonical assets linked to their connection, matched RepoSyncAsset, and triggered operation without exposing raw payload/result/error bodies, connection-level delivery health summaries, local callback rehearsal readiness previews, replay controls, secret rotation, basic rate limiting, delivery-id deduplication, deployment-manifest `ASSOPS_GATEWAY_URL` wiring, and public webhook URL normalization to an HTTP(S) origin.

Recommended next slice:

- Run provider-level callback rehearsals against real Gitea/GitHub webhook settings once a public staging hostname is available; the current readiness preview intentionally performs no external provider calls.

### 4. Project Template Instantiation Partially Complete

The Notion v0.4 goal is `project.create_from_template`. The code now has a read-only `project_templates` table, seeded built-in template metadata, list/detail/preview APIs, UI catalog/detail/preview modals, asset inventory exposure, provider account management for GitHub/Gitea template provisioning, manual provider token-env rotation, automated token-env rotation execution for ready safe candidates, manual provider token/API health checks, sanitized external provider provisioning diagnostics in template run results, token rotation due/soon/fresh visibility, and account-level rotation planning without exposing raw env names. The automated rotation path only updates ready provider account token-env metadata, refreshes linked git remote metadata, syncs canonical assets, and returns sanitized accounts; it does not read token values or call provider APIs. A durable `project.create_from_template` operation persists `project_template_runs` step status. Project template runs are now canonical assets linked to their project, source template, operation run, and produced starter-file assets without exposing raw run input/result/error payloads. The operation creates the project asset, repository metadata, template-defined Git remotes, a RepoSyncAsset when source/target remotes can be resolved by ID or template remote key, and starter file rows exposed as `template_file` assets. When a template remote is explicitly configured with `provider_type: local_bare` and an absolute local `remote_url`, the worker initializes a bare Git repository, commits the starter files, pushes the default branch, and marks the repository/files as provisioned. When a template remote is configured with `provider_type: github` or `provider_type: gitea`, a provider account reference, and an allowed token environment variable, the worker can create the upstream repository before pushing starter files. Template remotes marked `protected` create the external repository but skip starter-file pushes by default unless metadata explicitly sets `allow_protected_branch_push`; protected remotes can now include a `branch_strategy` such as `proposed_branch` to surface a safe proposed branch and provider review next action without writing to the protected default branch, plus a `provider_review_readiness` gate and nested provider review execution plan. That plan now prepares an approval-ready execution request payload summary for branch creation, starter-file commit, and PR/MR creation while still marking provider API mutation disabled, redacting payload details, confirming no token is included, and exposing provider review execution guardrail gates for the config switch, provider API adapter, branch refs, and staged starter-file payload. Approval requests now include a content-redacted starter-file payload summary with file count, id, path, kind, and status only; file content stays out of approval payloads and execution results. They also include a redacted provider API request plan for branch ref creation, starter-file commit, and PR/MR opening with endpoint keys, methods, payload-shape names, file counts, and no URL, token, or file content, plus a provider review preflight reconciliation summary for credential configured/present booleans, adapter status, gate status, blocked reasons, planned provider operations, a redacted adapter rehearsal summary for operation readiness and mutation-arming candidacy, a redacted mutation arming plan for config/rehearsal/mutation-gate readiness, a redacted adapter execution blueprint for future request builder/response handler/idempotency ledger implementation, a redacted execution target summary that defines the future adapter input boundary, and a redacted adapter contract listing required capabilities/scopes. Operators can request an operation approval from the Template Runs table, and approved requests record the same guardrail and preflight reconciliation result audit-only without provider API mutation; Approval audit details now expose a sanitized provider review approval summary with request, guardrail, credential preflight, starter-file staging, API plan, target summary, adapter contract, adapter rehearsal, mutation arming plan, adapter execution blueprint, preflight reconciliation, and result fields while still withholding the raw request payload. `ASSOPS_ENABLE_PROVIDER_REVIEW_EXECUTION=true` records that execution was requested, and `ASSOPS_ARM_PROVIDER_REVIEW_MUTATION=true` can mark a rehearsed mutation plan armed in the generated approval/execution audit, but provider API mutation remains disabled until a real adapter execution layer is added. External repositories that already exist also skip starter-file pushes by default unless metadata explicitly sets `allow_existing_repository_push`, so first-version runs surface a structured `repository_reconciliation` summary instead of writing into an unmanaged repository. The Template Runs table now shows those reconciliation states, provider review readiness, provider review execution request status, guardrail gates, redacted provider API plan status/endpoint keys, mutation arming status, and operator next-step guidance directly instead of requiring operators to inspect raw JSON, while keeping hard run failures separate in the error column. Backward-compatible inline provider metadata still works for existing templates. Failed or unprovisioned runs that already created project metadata can now enqueue an operator-triggered repository provisioning retry through the control worker, preserving the original run audit trail while reusing the existing project, repository, remotes, and starter files. This is still a first-pass provider create path, not a full automated provider reconciliation system.

The older `git_remotes.source_provider_id` column remains an unused provider-layer placeholder; current provider account wiring uses `source_account_id`.

Provider review `response_diagnostics` is intentionally a redacted audit contract: it lists planned operation endpoint keys, success/retry status classes, request-id/rate-limit/error-code diagnostic fields, and pending status only. It must not persist provider response bodies, headers, tokens, provider URLs, or repository references until a later adapter layer adds separately reviewed storage and redaction rules. Provider review `idempotency_plan` is also redacted: it records only operation-level replay checks, retry/conflict policies, and the requirement for a persisted attempt ledger; it does not expose raw idempotency keys, branch names, repository refs, provider URLs, tokens, or file content. Provider review `target_summary` is the corresponding redacted adapter input boundary: it records provider/review kind, validated branch refs, staged file count, planned endpoint operations, and adapter mutation-blocked status without provider URLs, repository refs, tokens, raw idempotency material, or file content. GitHub pull request and Gitea merge request adapter contracts are now registered as planned capabilities for branch creation, starter-file commit, and review-request opening, and `adapter_rehearsal` derives a redacted ready/blocked operation summary plus mutation-arming candidate signal from request-envelope readiness and credential preflight only. `mutation_arming_plan` now derives a redacted blocked/ready-to-arm/armed state from the execution config, rehearsal readiness, and `ASSOPS_ARM_PROVIDER_REVIEW_MUTATION` gate, while explicitly suppressing provider URLs, repository refs, tokens, file content, and provider API call/mutation flags in approval audit output. `execution_blueprint` adds the next redacted adapter boundary: operation-level payload builder, response handler, provider client, and idempotency ledger requirements, still with request/response bodies, headers, provider URLs, repository refs, tokens, branch names, file content, and live provider mutation disabled. Provider API mutation remains disabled until the real adapter execution layer is separately implemented and reviewed. Approved provider review execution now creates a local `provider_review_attempts` ledger entry per planned branch/commit/review operation, including the create-branch -> commit-files -> open-review operation order, dependencies, dependency status, redacted request builder/response handler/idempotency requirements, redacted response status/retry diagnostics, and a redacted orchestration summary with next-operation/count signals plus an execution-candidate gate set for the next operation, still with provider API calls disabled and attempt summaries redacted, so future adapters have a durable retry/idempotency/orchestration anchor before any external mutation is enabled. Execution-candidate gates are explicitly categorized as `data_integrity` metadata checks or `execution_blocker` gates; green metadata gates do not authorize provider mutation while adapter and mutation-arming gates remain blocked. Attempt response status is allowlisted to `pending`, `success`, `retryable`, `failed`, and `blocked`; unknown statuses are redacted as `blocked`, and endpoint keys/status classes are allowlisted before display. Approval audit details also read the persisted ledger from `provider_review_attempts` as a redacted fallback, which keeps operator review evidence visible even when the original approval result payload predates the ledger summary.
